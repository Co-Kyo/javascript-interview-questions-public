# LRU 缓存 - 四步讲解

## 第一步：理解问题本质

### 什么是 LRU？

LRU（Least Recently Used）是一种缓存淘汰策略：**当缓存满时，优先移除最久没有被访问的数据**。

用一个生活类比：

> 想象你书桌上只能放 3 本书。每次你拿起一本书阅读后，把它放到桌面最右边。当桌面满了又需要放新书时，把最左边的那本（最久没碰的）收走。

### 核心操作

| 操作 | 说明 |
|------|------|
| `get(key)` | 查找 key → 找到则返回值并**标记为最近使用** → 找不到返回 -1 |
| `put(key, value)` | 插入/更新 → 如果满了则**先淘汰最久未使用的** → 再插入 |

### 关键约束

- **两个操作都必须是 O(1)**——不能每次遍历整个缓存
- 这意味着我们需要**同时具备**：快速查找（哈希表）+ 维护顺序的能力（链表）

## 第二步：选择数据结构

### 为什么需要两种数据结构的组合？

| 需求 | 单独用 HashMap | 单独用链表 | HashMap + 双向链表 |
|------|:-:|:-:|:-:|
| O(1) 查找 | ✅ | ❌ O(n) | ✅ |
| O(1) 插入/删除 | ✅ | ✅（已知节点） | ✅ |
| 维护使用顺序 | ❌ 无序 | ✅ | ✅ |

**答案**：HashMap 用来 O(1) 找到节点，双向链表用来 O(1) 维护使用顺序。

### 结构示意

```
HashMap (key → node)
┌─────┐
│  1  │──→ [Node1] ↔ [Node2] ↔ [Node3]  ← 双向链表
│  2  │──→    ↑                        ↑
│  3  │──→  head                      tail
└─────┘
      最久未使用 ←──────────────→ 最近使用
```

## 第三步：拆解核心操作

### get(key) 的流程

```
1. 从 HashMap 中查找 key
2. 如果不存在 → 返回 -1
3. 如果存在 → 获取 node
4. 将 node 从链表当前位置「摘除」
5. 将 node 插入到链表尾部（标记为最近使用）
6. 返回 node.value
```

### put(key, value) 的流程

```
1. 从 HashMap 中查找 key
2. 如果已存在：
   a. 更新 node.value
   b. 将 node 移到链表尾部
3. 如果不存在：
   a. 检查容量是否已满
   b. 如果满了：
      - 删除链表头部节点（最久未使用）
      - 从 HashMap 中删除对应 key
   c. 创建新节点
   d. 插入链表尾部
   e. 写入 HashMap
```

### 为什么节点需要存储 key？

淘汰时我们需要同时从**链表和 HashMap** 中删除。链表中删除节点很简单，但要从 HashMap 中删除就需要知道 key 是什么。所以节点必须存储 `key`。

> 💡 **哨兵节点技巧**：链表实现中建议使用 `head` / `tail` 两个哨兵（dummy）节点，使插入和删除操作统一化，无需判断链表是否为空或操作的是否是头尾节点：
>
> ```javascript
> // head.next → 最久未使用节点；tail.prev → 最近使用节点
> _addToTail(node) {
>   node.prev = this.tail.prev;
>   node.next = this.tail;
>   this.tail.prev.next = node;
>   this.tail.prev = node;
> }
> _removeNode(node) {
>   node.prev.next = node.next;
>   node.next.prev = node.prev;
> }
> ```
>
> 这是链表题目的高频技巧，面试手写版必用。

## 第四步：对比两种方案 & 面试建议

### 方案对比

| 维度 | Map 版 | 双向链表 + HashMap 版 |
|------|--------|----------------------|
| 代码量 | ~30 行 | ~80 行 |
| 时间复杂度 | O(1) | O(1) |
| 空间复杂度 | O(n) | O(n) |
| 面试价值 | 展示对 Map API 的熟练度 | 展示底层数据结构功底 |
| 工程实用性 | ✅ 直接用，生产级 | 理解原理，面试考察 |

### ES6 Map 的关键特性

- `Map` 保证**插入顺序**，`keys().next().value` 可以拿到最早插入的 key
- `delete` + `set` 相当于把 key「移到最新」
- 这是 Map 版 LRU 能工作的根本原因

### 面试建议

1. **先写 Map 版**：快速给出正确解法，展示编码能力（2-3 分钟）
2. **再写手写版**：展示底层功底，画图解释双向链表的操作（8-10 分钟）
3. **讨论变体**：主动提及分片 LRU、TTL 过期、LRU vs LFU，展示工程视野
4. **手写版必画图**：面试中画出链表结构 + 操作过程，比干讲清晰 10 倍

### 常见错误

- ❌ 忘记节点存储 key → 淘汰时无法从 HashMap 中删除
- ❌ 手写版没有用哨兵节点 → 边界条件容易出错
- ❌ get 时忘记把节点移到尾部 → 使用顺序不准，淘汰逻辑错误
- ❌ put 已存在 key 时没有先删除再插入 → Map 版中顺序不更新

## 附录：边界情况速查

| 场景 | 预期行为 | 易错点 |
|------|----------|--------|
| capacity = 1 | 每次 put 都淘汰前一个元素 | 新手容易忘记处理 size 判断 |
| 空缓存 get | 始终返回 -1 | 无，但面试时需主动提及 |
| put 更新已有 key | 更新 value + 移到最新，不触发淘汰 | Map 版必须先 delete 再 set |
| 连续 get 同一个 key | 每次都移到最新位置 | 重复操作无副作用 |
| put 后立即 get | 返回刚写入的值 | 确认 put 正确更新了顺序 |
